# 49. Context API

Tag: **[BOTH]**

This topic is very important because it solves a **real architecture problem in React applications**.

---

# 1. Problem — Why Context API Exists

Imagine a typical React component tree.

```
App
 ├── Navbar
 │     └── UserAvatar
 ├── Sidebar
 │     └── Menu
 └── Dashboard
       └── UserProfile
```

Suppose we have **logged-in user data**.

Example:

```
user = {
  name: "Tanish",
  role: "admin"
}
```

Now we want to use this data in:

- Navbar
- UserAvatar
- UserProfile

---

### Traditional Solution (Props)

We pass props.

```
App
 └── Navbar user
      └── UserAvatar user
```

And also:

```
App
 └── Dashboard user
      └── UserProfile user
```

Code becomes:

```
<App user={user} />
```

Then:

```
<Navbar user={user}/>
<UserAvatar user={user}/>
```

This continues for **every level**.

---

# Problem: Prop Drilling

This situation is called **Prop Drilling**.

Definition:

Prop drilling means **passing props through many intermediate components that don't actually need them**.

Example:

```
App → Layout → Navbar → UserAvatar
```

But only `UserAvatar` needs the user.

Still we pass through:

```
Layout
Navbar
```

This causes:

- messy code
- difficult maintenance
- unnecessary dependencies

---

# Solution — Context API

Context API allows **sharing data globally across components without passing props manually at every level**.

Think of it like:

```
Global Data Store
```

Any component can access the data.

---

# Real World Analogy

Imagine **WiFi router**.

Instead of connecting cables from room to room, every device connects directly to the router.

```
Router (Context Provider)
     ↓
Devices (Components)
```

---

# Core Idea of Context API

React Context has **three main parts**.

```
1 Create Context
2 Provide Context
3 Consume Context
```

Or conceptually:

```
Context
Provider
Consumer
```

---

# Visual Architecture

```
Context
   ↓
Provider
   ↓
Component Tree
   ↓
Any component can read data
```

Example:

```
UserContext
      ↓
UserProvider
      ↓
App
 ├ Navbar
 │   └ Avatar
 └ Dashboard
     └ Profile
```

All components can access user.

---

# Step 1 — Create Context

React provides a function:

```
createContext()
```

Example:

```
UserContext.js
```

```javascript
import { createContext } from "react";

/*
Create a new context object.

This object will hold shared global data.
*/
export const UserContext = createContext();
```

Now we created a **global container**.

But it does not yet provide data.

---

# Step 2 — Provider

Provider supplies data to the component tree.

Example:

```
UserProvider.jsx
```

```javascript
import { UserContext } from "./UserContext";

/*
Provider component responsible
for supplying global data.
*/

function UserProvider({ children }) {
  // Global shared state
  const user = {
    name: "Tanish",
    role: "Admin",
  };

  return (
    /*
    Provider makes data available
    to all nested components
    */

    <UserContext.Provider value={user}>
      {/*
      children represents
      all nested components
      */}

      {children}
    </UserContext.Provider>
  );
}

export default UserProvider;
```

Important:

```
value={user}
```

This is the **global data**.

---

# Step 3 — Wrap App

Now we wrap the application.

```
main.jsx
```

```javascript
import React from "react";
import ReactDOM from "react-dom/client";
import App from "./App";
import UserProvider from "./UserProvider";

/*
Wrap the entire application
with the context provider.
*/

ReactDOM.createRoot(document.getElementById("root")).render(
  <UserProvider>
    <App />
  </UserProvider>,
);
```

Now every component inside `App` can access user.

---

# Step 4 — Consume Context

Now a component can read the data.

Example:

```
UserProfile.jsx
```

We use **useContext hook**.

```javascript
import { useContext } from "react";
import { UserContext } from "./UserContext";

/*
Component reading global context data
*/

function UserProfile() {
  // Access shared context value
  const user = useContext(UserContext);

  return (
    <div>
      {/* Display user data */}
      <h2>User Name: {user.name}</h2>

      <p>Role: {user.role}</p>
    </div>
  );
}

export default UserProfile;
```

Now this component directly reads context.

No props required.

---

# Example Component Tree

```
App
 ├ Navbar
 │   └ Avatar
 └ Dashboard
     └ Profile
```

All components can access:

```
UserContext
```

without props.

# Complete Folder Example

```
src
 ├ context
 │   ├ UserContext.js

 │   └ UserProfile.jsx

# Example — Theme Context

Another common use case.

```

```javascript

```

### Theme Provider

```javascript
import { useState } from "react";
import { ThemeContext } from "./ThemeContext";

/*
Global theme provider
*/

function ThemeProvider({ children }) {
  // Theme state
  const [theme, setTheme] = useState("light");

  // Toggle function
  const toggleTheme = () => {
    setTheme(theme === "light" ? "dark" : "light");

  );
}

import { useContext } from "react";
import { ThemeContext } from "../context/ThemeContext";

function Navbar() {

  return (
    <div>
      <p>Current Theme: {theme}</p>

      <button onClick={toggleTheme}>Toggle Theme</button>
    </div>
  );
}

export default Navbar;
```

# When NOT to Use Context

Context is **not good for frequently changing data**.

Example:

```
chat messages
real time data
large state
```

Reason:

Every context update **re-renders all consumers**.

This can hurt performance.

Better alternatives:

```
Redux
Zustand
React Query
```

# Tradeoffs

### Advantages

1 Global state without props drilling

2 Cleaner component tree

3 Built into React

4 Easy to use

---

### Disadvantages

1 Performance issues with frequent updates

2 Hard to debug large contexts

3 Not ideal for complex state

# 50. useContext Hook

Tag: **[BOTH]**

This topic builds directly on the previous concept (**Context API**).
Previously we learned **how to create and provide global data**.
Now we learn **how components read that data**.

---

# 1. Problem — Why `useContext` Exists

Earlier versions of React accessed context using a **Consumer component**.

Example:

```javascript
<UserContext.Consumer>{(user) => <h2>{user.name}</h2>}</UserContext.Consumer>
```

Problems with this approach:

1. Nested syntax becomes messy
2. Hard to read
3. Not ergonomic in functional components
4. Multiple contexts create callback pyramids

Example of bad nesting:

```
ThemeContext.Consumer
   └ UserContext.Consumer
        └ SettingsContext.Consumer
```

React introduced **Hooks** to solve this.

One of them is:

```
useContext()
```

---

# 2. What `useContext` Is

`useContext` is a **React Hook that allows functional components to read context values directly**.

Instead of:

```
Consumer component
```

We write:

```
const value = useContext(ContextName)
```

Simple mental model:

```
useContext(Context)
        ↓
Reads value from nearest Provider
```

---

# 3. Basic Syntax

```javascript
const value = useContext(MyContext);
```

Explanation:

| Part       | Meaning                                   |
| ---------- | ----------------------------------------- |
| useContext | React hook                                |
| MyContext  | context object created with createContext |
| value      | data provided by Provider                 |

---

# 4. Visual Flow

```
createContext()
      ↓
Provider (value)
      ↓
Component Tree
      ↓
useContext()
      ↓
Reads value
```

Example:

```
UserContext
      ↓
UserProvider value={user}
      ↓
App
 ├ Navbar
 │   └ Avatar
 └ Dashboard
     └ Profile
```

`Avatar` can call:

```
useContext(UserContext)
```

### Rule 1 — Must be inside Provider

Bad:

```

```

```

```

---

### Rule 2 — React finds the nearest Provider

Example:

```

   ↓
Navbar
   ↓
ThemeProvider (light)
   ↓
Button
```

`Button` will receive:

```
light
```

because it is the **nearest provider**.

# 9. Performance Consideration

Important concept.

When **context value changes**, React **re-renders all consuming components**.

Example:

```
ThemeContext update
```

Every component using:

```

```

will re-render.

This is why **large applications sometimes avoid context for heavy state**.

# 51. useReducer Hook

Tag: **[BOTH]**

This hook introduces a **very important architectural pattern** used in complex React applications.

It is the **same idea used in Redux, Redux Toolkit, and many state libraries**.

---

# 1. Problem — Why `useReducer` Exists

React provides `useState` for managing state.

Example:

```javascript
const [count, setCount] = useState(0);
```

This works well for **simple state**.

Example use cases:

```
counter
toggle
input value
loading state
```

---

### But problems appear when state becomes complex.

Example:

```
shopping cart
form state
authentication
todo list
```

You start writing many state updates:

```
setCart()
setTotal()
setItems()
setDiscount()
```

State logic becomes **scattered across many places**.

Example:

```javascript
setItems([...items, newItem]);
setTotal(total + price);
```

Hard to maintain.

---

# Solution — Reducer Pattern

Instead of updating state everywhere, we create **one centralized state manager**.

Concept:

```
Action → Reducer → New State
```

Visual flow:

```
Component
   ↓
dispatch(action)
   ↓
Reducer function
   ↓
returns new state
   ↓
React updates UI
```

This pattern keeps **state logic predictable**.

---

# 2. Mental Model

Think of **Reducer like a machine**.

```
Input → Processing → Output
```

Example:

```
Action: INCREMENT
State: count = 5
Result: count = 6
```

Reducer decides **how state changes**.

---

# 3. Real Life Analogy

Imagine a **bank system**.

```
Action → deposit
Action → withdraw
Action → check balance
```

The bank logic decides what happens.

Similarly:

```
dispatch(action)
```

goes to reducer.

---

# 4. Syntax of useReducer

```javascript
const [state, dispatch] = useReducer(reducer, initialState);
```

Explanation:

| Part         | Meaning                            |
| ------------ | ---------------------------------- |
| reducer      | function controlling state changes |
| initialState | starting state                     |
| state        | current state                      |
| dispatch     | function to send actions           |

---

# 5. Reducer Function

Reducer is a **pure function**.

Structure:

```javascript
function reducer(state, action) {
  switch (action.type) {
    case "ACTION_NAME":
      return newState;
    default:
      return state;
  }
}
```

Important rule:

```
Reducer must never mutate state
```

Always return **new state**.

---

# 6. Basic Counter Example

### Step 1 — Reducer Function

```javascript
/*
Reducer function controls how state changes.
*/

function counterReducer(state, action) {
  /*
  action.type tells reducer
  what operation should happen
  */

  switch (action.type) {
    case "INCREMENT":
      /*
      Return new state
      */
      return { count: state.count + 1 };

    case "DECREMENT":
      return { count: state.count - 1 };

    default:
      return state;
  }
}
```

---

### Step 2 — useReducer Hook

```javascript
import { useReducer } from "react";

/*
Initial state of reducer
*/
const initialState = { count: 0 };

function Counter() {
  /*
  state = current state
  dispatch = function to send actions
  */

  const [state, dispatch] = useReducer(counterReducer, initialState);

  return (
    <div>
      <h2>Count: {state.count}</h2>

      <button onClick={() => dispatch({ type: "INCREMENT" })}>Increase</button>

      <button onClick={() => dispatch({ type: "DECREMENT" })}>Decrease</button>
    </div>
  );
}

export default Counter;
```

---

# 7. Dispatch Function

Dispatch sends **actions to reducer**.

Example:

```javascript
dispatch({ type: "INCREMENT" });
```

Action object structure:

```
{
   type: "ACTION_NAME",
   payload: optionalData
}
```

Example:

```
dispatch({
   type: "ADD_ITEM",
   payload: item
})
```

---

# 8. Example — Todo List Reducer

This is a **realistic example**.

---

### Initial State

```javascript
const initialState = {
  todos: [],
};
```

---

### Reducer

```javascript
function todoReducer(state, action) {
  switch (action.type) {
    case "ADD_TODO":
      /*
      Add new todo
      */

      return {
        todos: [...state.todos, action.payload],
      };

    case "DELETE_TODO":
      /*
      Remove todo
      */

      return {
        todos: state.todos.filter((todo, index) => index !== action.payload),
      };

    default:
      return state;
  }
}
```

---

### Component

```javascript
import { useReducer, useState } from "react";

function TodoApp() {
  const [state, dispatch] = useReducer(todoReducer, { todos: [] });

  const [input, setInput] = useState("");

  const addTodo = () => {
    dispatch({
      type: "ADD_TODO",
      payload: input,
    });

    setInput("");
  };

  return (
    <div>
      <input value={input} onChange={(e) => setInput(e.target.value)} />

      <button onClick={addTodo}>Add Todo</button>

      <ul>
        {state.todos.map((todo, index) => (
          <li key={index}>
            {todo}

            <button
              onClick={() =>
                dispatch({
                  type: "DELETE_TODO",
                  payload: index,
                })
              }
            >
              Delete
            </button>
          </li>
        ))}
      </ul>
    </div>
  );
}

export default TodoApp;
```

# 11. Combining Context + useReducer

This is **very common in production React apps**.

Pattern:

```
Context → global store
useReducer → state logic
```

Architecture:

```
AuthContext
   ↓
AuthProvider
   ↓
useReducer(authReducer)
   ↓
dispatch actions
```

Example:

```javascript
const AuthContext = createContext();

function AuthProvider({ children }) {
  const [state, dispatch] = useReducer(authReducer, initialState);

  return (
    <AuthContext.Provider value={{ state, dispatch }}>
      {children}
    </AuthContext.Provider>
  );
}
```

Now any component can:

```
dispatch actions globally
```
