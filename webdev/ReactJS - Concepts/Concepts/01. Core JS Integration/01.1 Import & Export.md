# IMPORT AND EXPORT IN REACT

(Problem → Concept → Folder → Code → Explanation → Output)

---

## 1. Problem Without Import & Export

If React did not support import/export:

* All code would be in one file
* No separation of concerns
* No modularity
* No reusability
* Huge unmanageable App.jsx

Example problem:
You create Header, Footer, Card, Button — but cannot place them in separate files.

Result:
One massive file with thousands of lines.

---

## 2. Core Concept

React uses **ES6 Modules** to split code into files.

Two key operations:

### Export

Makes something available to other files.

### Import

Brings that exported thing into another file.

This creates modular, clean architecture.

---

## 3. Types of Export

### 1. Default Export

Used for exporting ONE main thing from a file.

```js
export default ComponentName;
```

### 2. Named Export

Used when exporting MULTIPLE items.

```js
export { variable1, function2 };
```

## 5. Code Implementation

### A. Default Export Example

#### src/components/Header.jsx

```jsx
// Header.jsx
// Default export example.

function Header() {
  return <h2>This is Header Component</h2>;
}

export default Header;
```

Explanation:

* Only ONE main export.
* Import name can differ while importing.

---

### B. Import Default Export

#### src/App.jsx

```jsx
import Header from "./components/Header";

function App() {
  return (
    <div>
      <h1>Import & Export Concept</h1>
      <Header />
    </div>
  );
}

export default App;
```

---

## 6. Named Export Example

### C. Multiple Named Exports

#### src/components/Footer.jsx

```jsx
// Footer.jsx
// Named exports example.

export function Footer() {
  return <p>This is Footer Component</p>;
}

export function Copyright() {
  return <p>© 2025 React App</p>;
}
```

---

### D. Import Named Exports

#### src/App.jsx

```jsx
import Header from "./components/Header";
import { Footer, Copyright } from "./components/Footer";

function App() {
  return (
    <div>
      <h1>Import & Export Concept</h1>

      <Header />
      <Footer />
      <Copyright />
    </div>
  );
}

export default App;
```

## 7. Default vs Named Comparison

| Feature        | Default Export    | Named Export      |
| -------------- | ----------------- | ----------------- |
| Number allowed | One               | Multiple          |
| Import syntax  | any name          | must match        |
| Brackets       | No                | Yes               |
| Usage          | Primary component | Utility functions |

---

## 8. Common Mistakes

### ❌ Forgetting curly braces with named export

```js
import Footer from "./Footer";   // Wrong
```

✅ Correct:

```js
import { Footer } from "./Footer";
```

---

### ❌ Wrong name for named export

```js
import { Foot } from "./Footer";  // Error
```

Must match exactly.

---

## 9. How Import/Export Works Internally

Flow:

```
Component File → export → ES Module System → import → Used in App
```

Bundler (Vite / Webpack) resolves these modules automatically.

---

## 10. Why Import & Export Is Important

| Problem               | Import/Export Solution |
| --------------------- | ---------------------- |
| Single file chaos     | Modular code           |
| Unorganized structure | Structured project     |
| No reuse              | Component reusability  |
| Hard debugging        | Easy maintenance       |

---

## 11. Best Practices

* One component per file
* Use default export for main component
* Use named export for helpers
* Keep filenames matching component name
